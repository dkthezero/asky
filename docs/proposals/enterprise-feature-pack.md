# Proposal: Enterprise Feature Pack (P7)

**Target Audience:** Engineering orgs, platform teams, AI governance officers
**Status:** Planning — seeking PO review

---

## Executive Summary

agk v0.2.x serves individuals and small teams well. Enterprise customers (100+ developers, regulated industries) need governance, auditability, security, and orchestration features. This proposal adds 5 capabilities that transform agk from a personal tool into a team platform.

| # | Feature | Problem | Solution |
|---|---------|---------|----------|
| 1 | **Enterprise Policy & Compliance** | CISO wants to ban unverified skills; legal needs audit trails | Policy engine with denylists, mandatory sign-offs, compliance reports |
| 2 | **Team Config Synchronization** | Onboarding 50 devs means installing 20 skills each by hand | Org-level `.agk/team.toml` synced from central repo; auto-install on `agk sync` |
| 3 | **Skill Signing & Provenance** | Installing a skill from internet ≈ running `curl | bash` | GPG-signed skills; `agk verify --team-key` checks signatures before install |
| 4 | **Centralized Telemetry & Reporting** | PM asks "which skills does the frontend team actually use?" | Anonymized org telemetry export; per-team breakdown CSV; stale-skill report |
| 5 | **GitHub Enterprise Vaults + SSO** | Company uses GHES, not github.com | GHES vault adapter + `gh auth` token pass-through for private enterprise repos |

## 1. Enterprise Policy & Compliance

### Problem
- Risk: developer installs a skill that ships source code to an unknown URL
- No audit trail of who installed what, when
- No mechanism to enforce "only approved vaults"

### Solution
Policy engine driven by `~/.config/agk/policy.toml` (Global) + `.agk/policy.toml` (Workspace):

```toml
[policy]
# Block all skills except from approved vaults
allow_vaults = ["github.com/acme-org/*", "clawhub.ai/acme-org/*"]
prohibit_vaults = ["github.com/untrusted/*"]

# Require digital signature from acme-security team
require_signature = true
approved_signers = ["security@acme.com"]

# Skills that trigger mandatory review before install
review_required_patterns = ["network", "http", "api.*send"]

# Audit trail
audit_log = "~/.config/agk/audit.log"
```

### Behavior
- **TUI install:** If policy blocks, modal says: "This skill violates policy: blocked vault. Contact security@acme.com."
- **CLI install:** Exit code `4` (POLICY_VIOLATION), prints rule violated
- **Audit log:** Append-only log of every install/remove/enable action with timestamp, user, skill, vault

### Files
- `src/domain/policy.rs` — `PolicyConfig` struct, `PolicyEngine`
- `src/app/policy.rs` — `check_install_policy()`, `check_vault_policy()`
- `src/infra/audit.rs` — `AuditLogger` (append-only file writer)
- `src/cli/commands.rs` — hook policy check before every install

## 2. Team Config Synchronization

### Problem
- New hire runs `agk` → empty workspace. Needs 15 skills installed manually.
- Team standardized on a skill pack, but everyone has different versions.

### Solution
`.agk/team.toml` checked into the repo:

```toml
# Auto-generated by `agk team init — do not edit manually
[team]
name = "frontend-platform"
source = "github.com/acme-org/ai-workflows"
branch = "main"
refresh_interval = "24h"

[[require]]
identity = "acme-org/react-skills"
vault = "github.com/acme-org/ai-workflows"
scope = "workspace"
version_constraint = ">= 2.0.0"

[[require]]
identity = "acme-org/component-generator"
vault = "internal-vault"
scope = "workspace"
```

### Behavior
- `agk sync` reads `team.toml` → installs missing skills → updates outdated
- If team config diverges, `agk team diff` shows what's missing
- `agk team update` pulls latest team.toml from source vault
- Badge in TUI: `[Team] 14/15 required skills installed`

### Files
- `src/domain/team.rs` — `TeamConfig`, `TeamRequirement`
- `src/app/team_sync.rs` — `sync_team_config()`, `diff_team_config()`
- `src/cli/commands.rs` — `team` subcommand (init, diff, update)
- `src/tui/widgets/team_banner.rs` — status banner in TUI

## 3. Skill Signing & Provenance

### Problem
- Skills are arbitrary code. SHA10 detects *changes*, not *malicious intent*.
- No way to verify that `react-parser` skill was actually authored by the React team.

### Solution
GPG-based skill signing via `SKILL.md` frontmatter:

```markdown
---
name: react-parser
version: 1.2.3
signatures:
  - signer: security@acme.com
    fingerprint: A1B2C3D4...
    signed_at: 2026-05-01T00:00:00Z
---
```

### Behavior
- `agk sign --skill react-parser --key ~/.gnupg/acme-signing.asc` — adds signature block to SKILL.md
- `agk verify react-parser` — checks signatures against `policy.toml` approved_signers
- TUI: `[✓]` badge for signed skills, `[!]` for unsigned in enterprise mode
- Install blocked if `require_signature = true` and no valid signature

### Files
- `src/domain/signature.rs` — `SkillSignature`, `SignatureVerification`
- `src/infra/crypto.rs` — GPG wrapper (or `pgp` crate for pure Rust)
- `src/cli/commands.rs` — `sign` subcommand, `verify` flag
- `src/tui/widgets/skills.rs` — badge rendering

## 4. Centralized Telemetry & Reporting

### Problem
- Current telemetry is local-only per machine. Platform team can't see team adoption.
- Managers don't know which skills are dead weight (stale for 90 days).

### Solution
**(Privacy-preserving by design)**

`~/.config/agk/analytics.toml` gains opt-in team reporting:

```toml
[settings]
enabled = true
# Anonymize: strip username, machine name. Report only skill names + counts.
team_report = true
# Daily or weekly aggregation before upload
upload_interval = "24h"
# Endpoint (company-hosted, not cloud)
report_endpoint = "https://agk-analytics.acme.internal/api/v1/usage"
```

### Behavior
- Background scanner collects as before
- Every 24h, `report()` POSTs a JSON payload:
  ```json
  {
    "team_id": "frontend-platform",
    "report_period": "2026-05-01T00:00:00Z/2026-05-02T00:00:00Z",
    "skills": [
      {"name": "react-parser", "invocations": 150, "active_users": 8}
    ]
  }
  ```
- No PII. No source code. No command arguments. Just skill names and counts.
- `agk telemetry team-report` generates a CSV for managers (skill, team invocations, active users, stale)

### Files
- `src/domain/telemetry.rs` — add `team_report` settings
- `src/infra/telemetry/reporter.rs` — `TeamReporter`, anonymization, HTTP POST
- `src/cli/commands.rs` — `telemetry team-report` subcommand

## 5. GitHub Enterprise Vaults + SSO

### Problem
- Company runs GHES. agk's `github` vault only supports github.com.
- Enterprise users authenticate via SSO; can't use personal access tokens.

### Solution
Extend vault definition with `enterprise_url`:

```toml
[[vault]]
id = "acme-private"
type = "github"
url = "https://github.acme.internal/acme-org/ai-workflows"
enterprise_url = "https://github.acme.internal"
# Token from `gh auth status` or `GITHUB_TOKEN` env
```

### Behavior
- Vault adapter detects `enterprise_url` → uses GHES API base URL
- Respects `gh auth` SSO session if available
- Falls back to `GITHUB_TOKEN` env var
- Private repos work the same as public (assuming token has access)

### Files
- `src/infra/vault/github.rs` — add `enterprise_url` field, customize API base URL
- `src/infra/vault/mod.rs` — parse `enterprise_url` from vault config
- `src/app/ports.rs` — vault def gains optional `enterprise_url`

## Acceptance Criteria

1. [ ] Installing from a blocked vault exits with code `4` and shows policy rule
2. [ ] `agk sync` auto-installs all `team.toml` requirements within 30s
3. [ ] `agk verify --team-key` returns `exit(0)` for signed skills, `exit(5)` for invalid signatures
4. [ ] Telemetry team report CSV contains no PII (no usernames, no machine names)
5. [ ] GHES private repo can be added as vault and listed successfully
6. [ ] All features have `--json` output mode for CI integration
7. [ ] Policy enforcement works in both TUI and headless CLI
